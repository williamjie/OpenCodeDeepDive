# 第五篇: 工具系统（Tool System）架构

> **用人话说**：Agent 的"手和脚"。没有工具，Agent 只能聊天；有了工具，Agent 才能干活。

---

## 工具是什么？

工具是 Agent 与外部世界交互的接口。比如：

| 工具 | 干什么 | 用人话说 |
|---|---|---|
| `read` | 读取文件 | 看代码 |
| `write` | 写入文件 | 写代码 |
| `edit` | 编辑文件 | 改代码 |
| `grep` | 内容搜索 | 找代码 |
| `glob` | 文件模式匹配 | 找文件 |
| `bash` | 执行命令 | 跑脚本 |
| `websearch` | 网页搜索 | 查资料 |
| `webfetch` | 网页获取 | 抓网页 |
| `question` | 询问用户 | 问你问题 |
| `todowrite` | 创建管理 TODO | 管理任务 |
| `skill` | 加载 skill | 用技能包 |
| `apply_patch` | 应用补丁 | 批量改代码 |

---

## 工具的定义

每个工具由这些部分组成：

| 部分 | 用人话说 |
|---|---|
| `description` | 工具是干什么的 |
| `input` | 输入参数的格式（用 Schema 验证） |
| `output` | 输出结果的格式（用 Schema 验证） |
| `execute` | 实际执行逻辑 |

**用人话说**：就像定义一个函数——输入什么、输出什么、怎么算。

---

## 工具的注册与执行

### 双层注册

工具注册有两层：

1. **应用层**（ApplicationTools）— 全局注册的工具
2. **位置层**（Tools.Service）— 按目录注册的工具，优先级更高

**用人话说**：应用层是"公司标配"，位置层是"部门定制"。部门定制优先。

### 执行流程

```
Agent 请求 → LLM 解析 → 工具调用
  → ToolRegistry 查找 → 匹配注册名称
  → 权限检查
  → 输入解码（验证格式）
  → 执行
  → 输出编码 + 大小限制
  → 返回结果到 LLM
```

**用人话说**：就像去食堂打饭——先看菜单（工具列表），选一个（LLM 决定），验证饭卡（权限检查），打饭（执行），打包（输出编码），拿走（返回结果）。

### Settlement 流程

这是工具执行的核心流程：

```
1. 解析注册 → 找到有效的命名注册
2. 解码输入 → 如果格式不对，不执行
3. 调用工具 → 使用 runner 提供的上下文
4. 编码输出 → 如果格式不对，不返回成功
5. 投影输出 → 转换为 LLM 能理解的格式
6. 边界处理 → 过大内容截断到托管存储
```

**用人话说**：每一步都有验证，确保不会因为格式错误导致系统崩溃。

---

## 输出边界处理

工具输出可能很大，需要截断：

| 类型 | 处理方式 | 用人话说 |
|---|---|---|
| 文本部分 | 受限大小，超大的保存到托管存储 | 文本太长就存到别处 |
| 结构化元数据 | 保留不变 | 元数据不动 |
| 原生媒体 | 由生产者限制，保持原样 | 图片/视频保持原样 |
| 空内容 | 度量结构化输出 | 空的也要记录 |

**用人话说**：就像寄快递，太大的东西要分箱寄，不能一个箱子装所有东西。

---

## 工具注册规则

| 规则 | 用人话说 |
|---|---|
| 同一名称最新注册生效 | 后注册的覆盖先注册的 |
| 关闭注册只移除当前注册 | 卸载插件时，恢复前一个注册 |
| Location 注册优先于应用注册 | 部门定制优先于公司标配 |
| 调用时捕获生效的注册 | 一旦开始执行，注册变更不影响当前调用 |
| 失效调用被拒绝 | 注册被移除后，旧的调用会被拒绝 |

**用人话说**：就像插拔 USB 设备——插上就用，拔掉就不能用了，但不会影响其他设备。

---

## 一句话总结

工具系统是 Agent 的"手和脚"，通过双层注册管理工具，执行时有完整的验证和边界处理，确保不会因为格式错误导致系统崩溃。
