# 第一篇: OpenCode 架构总览与核心技术栈

> **用人话说**：OpenCode 是一个开源的 AI 编程助手，类似 Claude Code。它用 TypeScript 写的，跑在 Bun 上，最大的特点是用了 Effect 这个函数式编程框架。

---

## 项目全景

OpenCode 是一个**多端 AI 编程助手**，它不是一个单纯的命令行工具，而是一套完整的系统：

- **终端界面**（TUI）— 命令行交互，类似 Claude Code 的终端体验
- **桌面应用** — 用 Electron 包了一层，做成原生应用
- **Web 控制台** — 浏览器端管理界面
- **Web App** — 浏览器端应用
- **命令行**（CLI）— 命令行入口
- **VSCode 插件** — 编辑器集成
- **JavaScript SDK** — 编程接口

**用人话说**：同一个 AI 大脑，接了 7 个不同的"壳"，让用户在不同场景下都能用。

---

## 项目结构：25 个包的大拼图

OpenCode 采用 Monorepo 架构（一个代码仓库管所有包），用 Turborepo + Bun workspaces 管理。25 个包大致分 5 层：

```
┌─────────────────────────────────────────────┐
│  UI 层：终端 / 桌面 / Web / VSCode          │
├─────────────────────────────────────────────┤
│  应用层：Agent 管理、Provider 适配           │
├─────────────────────────────────────────────┤
│  核心层：领域模型、Schema、事件系统          │
├─────────────────────────────────────────────┤
│  插件层：Plugin SDK、扩展接口               │
├─────────────────────────────────────────────┤
│  基础设施：数据库、身份认证、服务器          │
└─────────────────────────────────────────────┘
```

**用人话说**：就像盖房子，底层是地基（数据库、服务器），中间是框架（核心模型、事件系统），上面是装修（UI、插件）。每一层只管自己的事，互不干扰。

## 核心技术栈

### TypeScript + Bun

- 运行时用 **Bun**（比 Node.js 快）
- 全 TypeScript，strict 模式（类型检查很严）
- 测试用 `bun test`
- Linter 用 `oxlint`（比 ESLint 快）

### Effect v4：最核心的技术选型

这是 OpenCode 最特别的地方。Effect 是一个函数式编程框架，提供了远超 async/await 的能力。

**用人话说**：普通代码写异步像排队走路，Effect 写异步像流水线——每个工序（yield*）自动从仓库拿工具，用完放回去，出了问题自动清理。

Effect 的核心概念：

| Effect 概念 | 类比 | 在 OpenCode 中干什么 |
|---|---|---|
| `Effect.gen` | async/await 的升级版 | 所有异步逻辑编排 |
| `Context.Service` | 中央仓库 | 服务定义与获取 |
| `Layer` | 服务实现的"配方" | 服务怎么造出来的 |
| `Schema` | 运行时的类型检查 | 验证数据格式对不对 |
| `Scope` | 健身房的储物柜 | 资源生命周期管理 |
| `SynchronizedRef` | 带锁的变量 | 并发状态管理 |

**典型的服务模式**（伪代码）：

```
1. 定义接口：我要什么能力
2. 注册 Service：给这个能力起个名字
3. 实现 Layer：怎么造这个能力
4. 使用：yield* Service → 自动注入，拿过来就用
```

### AI SDK

- **默认运行时**：Vercel AI SDK 的 `streamText` / `generateObject`
- **原生运行时**：`@opencode-ai/llm`（实验性功能）
- **支持的 Provider**：OpenAI、Anthropic、Google、xAI、OpenRouter 等

### Drizzle ORM + SQLite

- 用 Drizzle ORM 操作 SQLite 数据库
- 存储会话数据、配置信息

## 五个设计哲学

从 OpenCode 的设计文档中提炼出五个核心原则：

### 原则一：Core 薄层

> 核心层只管"是什么"，不管"怎么做"

核心包（Core）只包含：
- 领域 Schema 和类型错误（数据长什么样）
- 状态容器（数据放哪里）
- 事件定义（发生了什么事）
- 插件钩子合约（插件能干什么）

业务逻辑通过插件实现，核心不包含应用策略。

**用人话说**：核心就像一个空的框架，只定义了"这里有个插槽"，具体插什么由插件决定。

### 原则二：插件化

> 业务逻辑通过插件实现

Provider 特定的行为、认证逻辑、模型发现、生成行为——这些都通过插件来实现。

**用人话说**：核心系统不管"怎么调用 OpenAI"，这些事交给插件去做。核心只管"这里有个人想调 AI"。

### 原则三：可重放变换

> 配置变更和插件加载都是可重放的

服务是热重载设计的：更新是细粒度的、可观察的，不需要拆掉整个进程。

**用人话说**：改个配置不用重启整个系统，像换电池一样热插拔。

### 原则四：事件溯源

> 会话状态通过事件流记录

所有状态变更都记录为持久化事件流。需要回溯时，重放事件就能还原。

**用人话说**：像银行账本，每一笔交易都记着。想知道某个时刻的余额？把之前的交易重放一遍就知道了。

### 原则五：Location 作用域

> 服务实例按工作目录隔离

不同目录有独立的服务实例和状态。

**用人话说**：在项目 A 里开的 Agent，和项目 B 里的 Agent，互不干扰，各自管各自的。

## 配置发现机制

OpenCode 支持多级配置，**越具体的优先级越高**：

1. **全局配置**：`~/.config/opencode/opencode.jsonc`
2. **项目配置**：从当前目录往上找 `opencode.json` / `opencode.jsonc`
3. **`.opencode` 目录**：`{项目}/.opencode/opencode.jsonc`（最高优先级）

**用人话说**：全局配置是"默认值"，项目配置可以覆盖它，`.opencode` 目录里的配置优先级最高。

---

## 一句话总结

OpenCode 是一个用 TypeScript + Effect 构建的多端 AI Agent 系统，核心设计哲学是"薄核心 + 插件化 + 事件溯源"，通过分层架构实现了高度的可扩展性和可维护性。
