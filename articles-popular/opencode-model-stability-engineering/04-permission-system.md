# 第四篇: 权限系统 — 防止模型误操作

> **用人话说**：Agent 可能会"乱来"（比如删文件），权限系统就是防止它乱来的"安全锁"。

---

## 为什么需要权限系统？

LLM 可能会：

| 风险 | 用人话说 |
|---|---|
| 调用不该调用的工具 | 比如删除文件 |
| 读取敏感信息 | 比如密码文件 |
| 执行破坏性操作 | 比如大规模删除 |
| 在错误的时间执行正确的操作 | 比如半夜删库 |

---

## 三层权限评估模型

```
用户输入 Action + Resources
        │
        ▼
  ┌─────────────────┐
  │ Layer 1: Agent   │  Agent 级别的权限规则（内置规则）
  │ 权限规则         │  └── 默认：禁止所有
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ Layer 2: 用户    │  用户保存的权限规则（持久化）
  │ 保存规则         │  └── 保存在 permission_saved 表中
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ Layer 3: 评估    │  Wildcard 匹配 + 优先级合并
  │ 引擎             │  └── 找到最后匹配的规则 → 最终效果
  └────────┬────────┘
           │
     ┌─────┴──────┬──────┐
     ▼            ▼      ▼
   Allow        Ask    Deny
   (允许)      (询问)   (拒绝)
```

---

## Rule → Effect 映射

每个权限规则由三个字段定义：

| 字段 | 用人话说 |
|---|---|
| `action` | 动作模式（支持 wildcard） |
| `resource` | 资源模式（支持 wildcard） |
| `effect` | "allow" / "deny" / "ask" |

**用人饭店**：就像"门禁规则"——谁能进（action）、进哪个门（resource）、允许/拒绝/询问（effect）。

---

## 核心评估引擎

`evaluate()` 函数是权限判断的核心：

```
合并所有规则集 → 找到最后匹配的规则 → 返回效果
如果没有匹配规则 → 默认 ask（询问用户）
```

### 关键设计决策

| 设计 | 用人饭店 |
|---|---|
| **`findLast`（而非 `findFirst`）** | 后面的规则覆盖前面的，方便在配置中追加覆盖规则 |
| **Wildcard 匹配** | `*` 匹配所有，支持 `fs.read.*` 等模式 |
| **默认 `ask`** | 没有匹配规则时，询问用户而不是静默允许或拒绝 |

**用人饭店**：就像法律——新法优于旧法，特别法优于普通法，没有法律规定就问法官。

---

## Deny 优先原则

```
首先检查是否有 deny 规则（Deny 优先）
然后合并保存的规则
三个资源中任何一个 deny → deny
任何一个 ask → ask
全 allow → allow
```

**Deny 优先于 Allow**：即使某些资源 allow，只要一个资源 deny，整个请求就被拒绝。

**用人饭店**：就像安检——只要有一件违禁品，整个包裹都不能过。

---

## 异步 Ask 流程

当权限评估结果为 "ask" 时，系统进入异步等待用户批准的模式：

```
权限评估 → ask → 存储到 pending map → 发布 Ask 事件 → 等待用户回复
```

用户通过 `reply()` 进行响应：

| 回复 | 用人饭店 |
|---|---|
| "reject" | 拒绝（可包含反馈） |
| "always" | 允许并记住 |
| "once" | 仅本次允许（不持久化） |

### 级联拒绝

当用户拒绝一个请求时，同 session 中所有 pending 请求也被级联拒绝：

```
用户拒绝请求 A → 同 session 的请求 B、C 也被拒绝
```

**用人饭店**：就像"连坐"——一个人犯错，同组的人都受影响。

### 智能 Auto-Allow

当用户选择 "always" 时，系统不仅保存当前规则，还会自动批准后续匹配的相同请求：

```
用户选择 "always" → 保存规则 → 扫描同 session 的 pending 请求 → 符合新规则则自动批准
```

**用人饭店**：就像"白名单"——一旦加入白名单，以后就不用再问了。

---

## 错误类型体系

| 错误类型 | 用人饭店 | 恢复策略 |
|---|---|---|
| `RejectedError` | 用户拒绝 | 尝试其他方案 |
| `CorrectedError` | 用户拒绝并给出反馈 | 根据反馈修正后重试 |
| `DeniedError` | 规则拒绝 | 不可恢复 |

**用人饭店**：就像三种"拒绝方式"——"不行"、"改成这样就可以"、"绝对不行"。

---

## 一句话总结

权限系统通过三层评估（Agent 规则 → 用户规则 → 评估引擎），实现 Deny 优先、Wildcard 匹配、异步询问，防止模型误操作。
