# 第一篇: 重试与回退 — 应对 LLM API 不可靠

> **用人话说**：LLM API 经常挂，OpenCode 有一套智能重试机制，知道什么时候该重试、什么时候该放弃。

---

## 为什么需要重试？

LLM API 是 AI Agent 的命脉，却也是最不可靠的环节：

| 问题 | 用人话说 |
|---|---|
| 服务器故障 | 5xx 错误、超时 |
| 限流（Rate Limit） | 429 Too Many Requests |
| 额度耗尽 | 免费层级/付费账户余额不足 |
| 过载 | Provider overloaded |
| 偶发错误 | 连接重置、DNS 解析失败 |

---

## 核心参数

| 参数 | 值 | 用人话说 |
|---|---|---|
| 初始延迟 | 2 秒 | 第一次失败后等 2 秒 |
| 退避因子 | 2x | 每次重试等待时间翻倍 |
| 最大延迟（无头信息） | 30 秒 | 最多等 30 秒 |
| 最大延迟（有头信息） | 无上限 | 服务端说等多久就等多久 |

---

## 延迟计算：三种模式

### 模式 1：服务端指定 retry-after-ms

```
服务端说"等 5000 毫秒" → 直接用这个值
```

最高优先级，直接使用服务端指定的毫秒等待时间。

### 模式 2：服务端指定 retry-after

```
服务端说"等 120 秒" → 用 120 秒
服务端说"等 Wed, 21 Oct 2026 07:28:00 GMT" → 计算时间差
```

同时兼容秒数和 HTTP 日期两种格式。

### 模式 3：指数退避（无头信息时）

```
2s → 4s → 8s → 16s → 30s（封顶）
```

**用人饭店**：就像打电话占线——等 2 秒没人接就等 4 秒，再没人就等 8 秒，最多等半分钟。

---

## 错误分类引擎：什么值得重试？

不是所有错误都值得重试。OpenCode 有精确的分类逻辑：

| 错误类型 | 是否重试 | 用人话说 |
|---|---|---|
| ContextOverflowError | ❌ 不重试 | 上下文溢出是设计问题，不是临时故障 |
| FreeUsageLimitError | ❌ 不重试 | 免费额度用完了，重试也没用 |
| GoUsageLimitError | ❌ 不重试 | 付费额度用完了，等恢复时间 |
| Overloaded | ✅ 可重试 | 服务器过载，等一会可能好 |
| 普通 APIError (5xx) | ✅ 可重试 | 服务器临时故障 |
| Rate Limit | ✅ 可重试 | 限流了，等一会就好 |

### 关键设计决策：ContextOverflowError 不重试

```
上下文溢出 → 重试只会浪费更多 token → 应该压缩而不是重试
```

**用人饭店**：就像行李箱满了——再塞也塞不下，应该整理一下而不是硬塞。

### 5xx 自动重试策略

即使 Provider SDK 没有标记某个错误为可重试的，只要状态码 >= 500，就强制重试。

**用人饭店**：宁可重试也不要静默失败——5xx 是服务器的问题，等一会可能就好了。

---

## 限流 UpSell：业务与工程的无缝衔接

OpenCode 在检测到免费额度耗尽时，不是简单地报错，而是触发一套 UpSell 流程：

```
免费额度用完 → 显示"订阅 OpenCode Go" → 引导用户付费
付费额度用完 → 显示"额度将在 X 时间后恢复" → 引导用户等待或升级
```

**用人饭店**：就像健身房——免费次数用完了，引导你办卡。

---

## Effect Schedule 集成

`policy()` 函数是整个重试策略与 Effect 函数式编程的桥梁：

```
重试策略 + Effect Schedule → 类型安全的重试调度器
```

**设计亮点**：Effect 的 `Schedule` 结合 `Cause.done()` 来终止重试序列，当 `retryable()` 返回 `undefined` 时，重试自动停止。

**用人饭店**：就像一个智能闹钟——该响的时候响，不该响的时候自动关闭。

---

## 一句话总结

重试系统通过智能错误分类、指数退避、服务端头信息解析，实现了"该重试时重试，该放弃时放弃"的策略，同时将技术错误转化为业务增长机会（UpSell）。
