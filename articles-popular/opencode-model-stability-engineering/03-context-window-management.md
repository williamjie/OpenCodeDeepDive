# 第三篇: 上下文窗口管理 — 防止 Agent 记忆溢出

> **用人话说**：Agent 的"记忆"有限，记太多就装不下了。OpenCode 有一套智能压缩机制，自动"整理笔记"。

---

## 为什么需要上下文管理？

LLM 有固定的上下文窗口限制（通常 8K-200K tokens）。当 Agent 持续运行时：

| 问题 | 用人话说 |
|---|---|
| 历史对话不断累积 | 聊天记录越来越多 |
| 工具执行结果占用大量 token | 工具返回的内容占空间 |
| 编码后的文件内容占比极大 | 读文件的内容占空间 |
| LLM 在处理长上下文时质量下降 | 记太多反而记不住 |

---

## 三层架构

```
┌─────────────────────────────────────────────────────────┐
│ Layer 1: 溢出检测                                         │
│ "是否需要做什么？"                                       │
├─────────────────────────────────────────────────────────┤
│ Layer 2: 可用 Token 计算                                  │
│ "还剩多少空间？"                                         │
├─────────────────────────────────────────────────────────┤
│ Layer 3: 自动压缩                                        │
│ "如何腾出空间？"                                         │
└─────────────────────────────────────────────────────────┘
```

---

## 可用 Token 计算

`usable()` 函数计算当前可用的输入 token 数量：

| 参数 | 用人话说 |
|---|---|
| `model.limit.context` | 模型的总上下文限制 |
| `model.limit.input` | 模型的输入限制 |
| `COMPACTION_BUFFER` | 保留 20K token 给输出 |

**计算逻辑**：

```
如果有 input_limit → input_limit - 保留量
如果没有 input_limit → 总上下文 - 输出预算
```

**用人饭店**：就像计算行李箱剩余空间——先减去必须留的空间（输出用的），剩下的才能装东西。

### 关键设计

- 始终为模型输出保留 token 空间（避免输出被截断）
- 支持两种限流模式：`model.limit.input`（输入限制）和 `model.limit.context`（总上下文限制）
- 预留 token 可配置

---

## 溢出检测

`isOverflow()` 决定是否已经溢出：

```
自动压缩关闭 → 不过问
无限上下文模型 → 不过问
当前 token >= 可用 token → 溢出
```

**精确计量**：`count` 包含了 prompt caching 的读写 token，这使得估算更加准确。

**用人饭店**：就像检查行李箱是否超重——把所有东西称一下，超过限制就超重了。

---

## 自动压缩引擎

### 压缩触发点

| 触发时机 | 用人话说 |
|---|---|
| `compactIfNeeded()` | 每次 LLM 请求前检查 |
| `compactAfterOverflow()` | LLM 返回溢出错误后被动压缩 |

### 压缩配置

| 参数 | 值 | 用人话说 |
|---|---|---|
| auto | true | 默认开启自动压缩 |
| buffer | 20,000 | 保留 20K token buffer |
| tokens | 8,000 | 压缩后保留 8K token 的最新历史 |

---

## 核心算法：select()

`select()` 函数实现"保留最新 N tokens"的逻辑：

```
1. 过滤掉已存在的压缩消息，只保留真实对话
2. 从后往前累加 token，找到在 tokens 预算内的分割点
3. 返回"历史部分"和"最新部分"
```

**用人饭店**：就像整理笔记——把旧的笔记（历史部分）总结成摘要，保留最近的详细内容（最新部分）。

---

## 让 LLM 总结 LLM

OpenCode 使用 LLM 自身来生成摘要，而不是使用简单的文本截断。

### 结构化摘要模板

压缩时，LLM 被要求生成包含 8 个固定段落的摘要：

1. **Goal**（目标）— 一句话总结任务
2. **Constraints & Preferences**（约束和偏好）
3. **Progress**（进度）— Done / In Progress / Blocked
4. **Key Decisions**（关键决策）
5. **Next Steps**（下一步）
6. **Critical Context**（关键上下文）
7. **Relevant Files**（相关文件）

**用人饭店**：就像项目周报——必须包含目标、进度、问题、下一步等固定内容。

### 安全设计

| 设计 | 用人饭店 |
|---|---|
| **禁止提及压缩过程** | 避免让 LLM 在回复中暴露"系统提示"（越狱保护） |
| **保留精确引用** | 文件路径、错误信息、命令等关键信息必须原样保留 |
| **合并而非替换** | 当已有摘要时，LLM 被要求"更新"而不是"重写" |

---

## 压缩执行流程

```
1. 安全检查：模型必须有上下文限制
2. 计算分割：select() 函数
3. 获取前次摘要（如果存在）
4. 构建提示词（更新模式 vs 创建模式）
5. 调用 LLM 生成摘要
6. 如果压缩失败，回退到不压缩（不阻塞主流程）
7. 发布压缩事件（可观测性）
```

**用人饭店**：就像整理笔记的流程——先检查是否需要整理，然后分割新旧内容，让 AI 总结旧内容，最后保存。

---

## 预检机制（节省 token）

`compactIfNeeded()` 在发送请求前先估算 token 用量：

```
如果估算结果 <= 可用量 → 跳过压缩（省 token）
```

**token 估算**：简单的 JSON 序列化后估算，不需要调用外部 tokenizer。

**用人饭店**：就像出门前先称行李——如果没超重就不用整理，省时间。

---

## 工具输出截断

为了防止工具输出消耗过多 token，工具输出被截断到 2000 字符：

```
工具输出 > 2000 字符 → 截断到 2000 字符 + "[truncated]"
```

**用人饭店**：就像快递包裹有大小限制——太大的包裹要分箱寄。

---

## 一句话总结

上下文管理通过三层架构（溢出检测、可用 token 计算、自动压缩），用 LLM 自身生成结构化摘要，防止 Agent 记忆溢出，同时保留关键信息。
