# 第七篇: 设计哲学与工程实践

> **用人话说**：总结一下 OpenCode 模型稳定性工程的设计思路和核心模式。

```
                    │    业务层防御            │

                    ┌──────────▼──────────────┐
                    │    运行时层防御          │
                    │  Retry、Max Steps、Fiber  │
                    └──────────┬──────────────┘
                               │

```

**用人饭店**：就像城堡的防御——有护城河、城墙、城门、卫兵，层层防御。

---

## Effect 函数式编程的稳定工程优势

| 特性 | 稳定性贡献 | 用人饭店 |
|---|---|---|
| **类型安全的错误** | 错误类型编译时检查 | 编译时就能发现错误 |
| **精确的错误捕获** | 通过 `_tag` 模式匹配 | 精确识别错误类型 |
| **Scope 管理** | 资源生命周期自动管理 | 用完自动清理，不会泄漏 |
| **Fiber 并发** | 结构化并发 | 不会产生"孤儿"任务 |
| **Cause 系统** | 完整记录错误链 | 不丢失任何错误上下文 |
| **不可变数据** | 状态变化可控 | 没有并发竞态 |

## 关键模式总结

### 模式：分层错误恢复（以上下文溢出为例）

```
Level 1: 预检（compactIfNeeded）→ 主动压缩 → 继续
Level 2: 溢出检测（isContextOverflowFailure）→ 被动压缩 → 继续
Level 3: 恢复失败（溢出恢复后再次溢出）→ die 报错 → 不可恢复
```

**用人饭店**：就像"三振出局"——第一次整理行李，第二次再整理，第三次就直接放弃。

### 模式：TurnTransition 非局部跳转

```
需要重新构建请求 → Effect.die(TurnTransitionError) → catchDefect → 重新构建
```

**用人饭店**：就像"紧急出口"——不管你在几楼，都能直接跳到一楼。

### 模式：Deny 优先评估

```
遍历所有规则 → 如果任何规则 deny → 立即拒绝
否则 → 遍历资源 → 存在 ask → 询问；全 allow → 允许
```

**用人饭店**：就像"一票否决"——只要有一票反对，就不能通过。

### 模式：智能的压缩摘要

```
对话历史 → 分割为 historical + recent
historical → LLM 生成结构化摘要
recent → 保持原始可读
下次压缩 → 合并历史摘要 + 最新历史 → 生成新摘要
```

**用人饭店**：就像"整理笔记"——把旧笔记总结成摘要，保留最近的详细内容。

## 可观测性事件

每个决策点都发布了结构化事件，支持监控和调试：

| 事件 | 触发点 | 包含信息 |
|---|---|---|
| `Compaction.Started` | 压缩开始 | sessionID, messageID, reason |
| `Compaction.Ended` | 压缩完成 | sessionID, summary, recent |
| `Tool.Failed` | 工具执行中断 | sessionID, callID, error type |
| `permission.v2.asked` | 权限询问 | requestID, action, resources |
| `permission.v2.replied` | 权限回复 | requestID, reply effect |

**用人饭店**：就像"监控摄像头"——每个关键操作都有记录，出了问题可以回溯。

## 对比总结

相比简单的 LLM API 调用封装，OpenCode 的模型稳定性工程体现了以下设计理念：

| 理念 | 用人饭店 |
|---|---|
| 1. **失败是常态，容错是默认** | 从 retry 策略到压缩恢复，系统默认所有操作都可能失败 |
| 2. **精准而非粗暴** | 错误分类不是简单的"可重试/不可重试"，而是精确分类 |
| 3. **LLM 辅助 LLM** | 用 LLM 生成摘要，用 Effect Schema 验证 LLM 输出 |
| 4. **安全深度防御** | 权限系统 + 工具 stale 检测 + deny 优先评估 |
| 5. **业务与工程融合** | 限流 UpSell 将技术错误转化为业务增长机会 |

**用人饭店**：就像一个"自愈系统"——出了问题能自动修复，修复不了就优雅降级，同时还能创造商业价值。

---

## 一句话总结

OpenCode 的模型稳定性工程体现了"失败是常态、容错是默认"的设计哲学，通过六层防御体系、Effect 函数式编程、可观测性事件，构建了一个健壮的 AI Agent 系统。
